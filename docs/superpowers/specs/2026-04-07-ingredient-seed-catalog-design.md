# Ingredient Seed Catalog — Design Spec

## Problem

The app's nutrition system works well once an ingredient is in the catalog,
but getting it there is tedious — each ingredient requires a USDA search,
careful selection from confusingly-named entries, and manual data entry.
The catalog ships with ~125 curated ingredients. Covering the breadth of
typical North American home cooking requires ~500, and the manual workflow
doesn't scale.

## Goal

Ship a high-quality seed catalog of ~500 common ingredients, each mapped to
a verified USDA FoodData Central entry with full nutrition, density, portions,
aisle, and aliases. Quality bar: every mapping is reviewed by a human.
No best-effort bulk imports — this is curated data.

## Non-Goals

- Per-kitchen customization workflow (the existing editor handles this)
- Automatic ingredient matching from recipe text (the existing resolver +
  inflector handles this)
- Replacing or modifying the existing `ingredient-catalog.yaml` format
- Building a permanent curation tool — the scripts are one-time-use

## Design

### Phase 1: Ingredient List

A categorized list of ~500 ingredient names, generated by Claude and edited
by the user. The list stands on its own — it represents the 500 most useful
ingredients for North American home cooking, independent of what's already
in the catalog. Existing catalog entries that overlap are a bonus, not the
starting point.

**Categories** (for organizational purposes during review only — the actual
aisle assignment happens in Phase 2):

- Produce (fruits, vegetables, fresh herbs)
- Dairy & Eggs
- Meat & Seafood
- Pantry (canned goods, dried beans/grains, pasta, rice, oils)
- Baking (flours, sugars, leaveners, chocolate, extracts)
- Spices & Seasonings (dried herbs, ground spices, blends)
- Condiments & Sauces
- Refrigerated (butter, tofu, fresh pasta, doughs)
- Frozen (vegetables, fruits, ice cream)
- Bread & Bakery
- Beverages & Alcohol (cooking wine, broth/stock)

**Naming conventions** (derived from the existing catalog):

- Capitalized natural language: "Heavy cream", not "heavy cream"
- Parenthetical qualifiers for variants: "Flour (all-purpose)", "Sugar (brown)"
- Parenthetical for form/packaging: "Tomatoes (canned)", "Chickpeas (canned)"
- Opinionated defaults: bare name = most common variant (e.g., "Milk" = whole)
- Nutritionally significant variants get separate entries (whole vs skim milk,
  white vs brown rice); trivial differences don't (salted vs unsalted butter
  is one exception — both are common enough to warrant entries)
- Cheese by name without suffix: "Cheddar", "Gruyère", not "Cheddar cheese"
- Singular or plural based on what reads naturally in a recipe
- Specific brands/products when that's what people actually buy:
  "Frank's Red Hot", "Worcestershire sauce"

**Deliverable:** A Markdown file with ingredients grouped by category. The user
edits this file directly — renaming, cutting, and adding entries — before
proceeding to Phase 2.

### Phase 2: USDA Search + AI Matching

A two-step process: a Ruby script fetches USDA search results, then Claude
Code subagents pick the best match for each ingredient.

#### Step 2a: USDA Search Script

A standalone Ruby script (`scripts/seed_catalog/usda_search.rb`) that:

1. Reads the ingredient list from Phase 1
2. For each ingredient, calls the USDA FoodData Central search API
3. Collects the top ~10 results per ingredient, preferring SR Legacy entries
4. Writes raw results to an intermediate JSON file

**Design decisions:**

- **Standalone** — no Rails dependency. Uses `net/http` and `json` only.
- **Resumable** — checks the output JSON for already-searched ingredients and
  skips them on restart.
- **Rate-limited** — small delay between USDA API calls.
- **Requires a USDA API key** — passed via environment variable or CLI arg.

**Output format** (`data/seed_catalog/usda_search_results.json`):

```json
[
  {
    "name": "Butter (unsalted)",
    "category": "Dairy & Eggs",
    "usda_results": [
      {
        "fdc_id": 173430,
        "description": "Butter, without salt",
        "dataset": "SR Legacy"
      },
      ...
    ]
  },
  ...
]
```

#### Step 2b: AI Pick (Claude Code Subagents)

Claude Code subagents read the search results JSON and, for each ingredient,
select the best USDA match. This runs inside Claude Code — no Anthropic API
calls, no SDK dependency.

The subagents work in batches (~20 ingredients per batch) and write their
picks back to the same JSON file, adding:

```json
{
  "ai_pick": {
    "fdc_id": 173430,
    "reasoning": "Standard unsalted butter, most common form"
  },
  "aisle": "Refrigerated",
  "aliases": ["Sweet cream butter"]
}
```

**Aisle values** are drawn from the app's fixed list: Fresh Bakery, Produce,
Gourmet, Bread, International, Health, Refrigerated, Alcohol, Cereal, Baking,
Spices, Snacks, Condiments, Pantry, Beverages, Frozen, Specialty, Personal,
Household, Miscellaneous.

**SR Legacy preference:** When multiple datasets have entries for the same
food, prefer SR Legacy for consistency with the existing catalog.

### Phase 3: Human Review

The user reviews the AI picks using an HTML review page generated from the
intermediate JSON.

#### HTML Review Page

A static HTML file (`data/seed_catalog/review.html`) generated by a script.
For each ingredient, it shows:

- Canonical name and category
- AI pick: USDA description, hyperlinked to
  `https://fdc.nal.usda.gov/food-details/{fdc_id}/nutrients`
- AI reasoning
- Top 3-4 alternatives, each hyperlinked to their USDA detail page
- Suggested aisle and aliases
- Status control (defaults to accept)

The user can click through to the USDA nutrient detail page for any entry
to verify the data looks right. This solves the core review problem: USDA
names are only interpretable in context of the alternatives, and sometimes
you need to see the actual nutrient numbers to pick the right entry.

#### Review Workflow

The review page includes lightweight JavaScript to track decisions in
`localStorage` and export them as a JSON file when the user is done.
Status values per ingredient:

- **accept** (default): Use the AI pick as-is
- **override**: Use a different FDC ID (from the alternatives or manually
  entered)
- **skip**: Exclude from the seed catalog
- **manual**: Flag for hand-curation later via the app's existing ingredient
  editor

The user can also edit the aisle and aliases inline.

### Phase 4: YAML Generation

A script reads the reviewed JSON and generates `ingredient-catalog.yaml`
entries for all accepted/overridden ingredients.

For each confirmed FDC ID, the script:

1. Fetches the full nutrient detail from the USDA API
2. Extracts nutrients, density, and portions using the same logic as
   `UsdaImportService` and `UsdaPortionClassifier`
3. Writes a YAML entry in the existing catalog format

**Output:** A new YAML file (`data/seed_catalog/seed_catalog.yaml`), not
appended to the existing catalog. The user diffs and merges at their
discretion. Existing catalog entries that overlap with seed entries are
flagged but not overwritten.

**Density and portions** follow the same extraction logic the app already
uses: `UsdaPortionClassifier` buckets USDA portions into density candidates
(volume-based) and portion candidates (discrete units like "slice", "large").
The best density is auto-selected; portions are included as-is.

## File Layout

```
scripts/seed_catalog/
  usda_search.rb          # Phase 2a: USDA search script
  generate_review.rb      # Phase 3: generates review.html from JSON
  generate_yaml.rb        # Phase 4: generates YAML from reviewed JSON

data/seed_catalog/
  ingredient_list.md      # Phase 1: categorized ingredient list (user-edited)
  usda_search_results.json # Phase 2a output, Phase 2b enriched
  review.html             # Phase 3: static review page
  seed_catalog.yaml       # Phase 4: final YAML output
```

All files under `scripts/seed_catalog/` and `data/seed_catalog/` are
one-time-use tooling. They do not ship with the app and can be deleted
after the seed catalog is merged into `ingredient-catalog.yaml`.

## Ambiguity Strategy

When a single ingredient name could map to multiple USDA entries with
meaningfully different nutrition profiles:

- **Bare name = most common variant.** "Milk" → whole milk. "Rice" → white
  long-grain. "Chicken breast" → raw, boneless, skinless.
- **Nutritionally significant variants get separate entries.** "Milk (skim)",
  "Rice (brown)", "Chicken thigh". The threshold is roughly: would a
  nutrition-conscious person care about the difference?
- **Trivially different variants don't.** "Cheddar" covers mild, sharp,
  and extra-sharp (nutritionally identical). One USDA entry suffices.
- **Form matters when it affects nutrition.** "Spinach" (raw) vs
  "Frozen spinach" — density and nutrient concentration differ enough
  to warrant separate entries.

## Dependencies

- USDA FoodData Central API key (free, already used by the app)
- Claude Code for the AI pick phase (no external API calls)
- Ruby standard library only for the scripts (no gems)

## What This Does NOT Change

- The existing `ingredient-catalog.yaml` format — seed entries use the
  identical schema
- The `IngredientCatalog` model, `IngredientResolver`, or any app code
- The existing USDA search/import workflow in the ingredient editor
- The nutrition calculation pipeline

The seed catalog is purely additive: more rows in the same YAML file,
consumed by the same `db/seeds.rb` loader.
